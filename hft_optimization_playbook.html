1. Branch prediction4 techniques▼

__builtin_expect — hint the likely path

You seeperf stat shows high branch-misses on an if-else in a tight loop
WhyCPU prefetches the likely branch. If your error-check is rarely true, tell the compiler.
// SLOW: CPU guesses 50/50
if (order.price <= 0) { handle_error(); }

// FAST: tell compiler this is rare
if (__builtin_expect(order.price <= 0, 0)) { handle_error(); }

// C++20: even cleaner
if (order.price <= 0) [[unlikely]] { handle_error(); }

Branchless conditional — eliminate the branch entirely

You seeBranch on random/unpredictable data (incoming market prices). No amount of hinting helps.
WhyIf data is truly random, the predictor WILL be wrong ~50%. Remove the branch.
// SLOW: branch on every tick
int best = (a < b) ? a : b;

// FAST: branchless min via arithmetic
int diff = a - b;
int best = b + (diff & (diff >> 31));

// ALSO: use cmov via std::min (compiler usually emits cmov)
int best = std::min(a, b); // check godbolt!

Lookup table — replace if-else chain with array index

You seeSwitch/if-else over message types, order types, or state machines
WhyArray lookup is one load. Switch is N compares. For small enums, table wins.
// SLOW: switch on message type
switch(msg.type) {
  case ADD: handle_add(msg); break;
  case MODIFY: handle_mod(msg); break;
  case DELETE: handle_del(msg); break;
}

// FAST: function pointer table (no branches)
using Handler = void(*)(const Msg&);
static constexpr Handler table[] = {handle_add, handle_mod, handle_del};
table[msg.type](msg); // single indirect call

Sort-and-process — help the predictor learn the pattern

You seeProcessing unsorted array with conditional: branch-misses ~45%
WhySorted data makes branches predictable. Classic interview question.
// After sorting, the branch predictor sees:
// false false false ... true true true
// It learns the transition point. Mispredict count drops to ~1.

SoA vs AoS — structure of arrays for hot fields

You seeL1 cache miss rate high when iterating over objects but only touching 1-2 fields
WhyAoS loads entire struct per cache line. If you only need price, 90% of the line is waste.

Cache-line padding — kill false sharing

You seeTwo threads, independent data, but perf c2c shows cache line bouncing
WhyTwo variables on the same 64B line = coherency traffic even if logically independent.

  alignas(64) std::atomic<size_t> head; // producer writes
  alignas(64) std::atomic<size_t> tail; // consumer writes
  // Now on separate cache lines. No bouncing.
};

3. Allocation elimination3 techniques▼

Arena / pool allocator — pre-allocate everything

You seemalloc/new appearing in perf flamegraph inside hot path
Whymalloc takes ~50-100ns. Your entire tick-to-order budget is 500ns. You can't afford it.
template<typename T, size_t N>
class Pool {
  alignas(T) char buf[sizeof(T) * N];
  T* free_list[N];
  size_t top = N;
public:
  Pool() { for (size_t i = 0; i < N; ++i) free_list[i] = reinterpret_cast<T*>(buf + i*sizeof(T)); }
  T* alloc() { return top ? free_list[--top] : nullptr; }
  void dealloc(T* p) { free_list[top++] = p; }
}; // O(1) alloc, O(1) dealloc, zero syscalls

4. Eliminate dynamic dispatch3 techniques▼

CRTP — static polymorphism via templates

You seeVirtual function call in hot path. Indirect jump = pipeline stall + icache miss.
// SLOW: virtual dispatch (~5-25ns overhead per call)
struct Strategy { virtual void on_tick(Tick&) = 0; };

// FAST: CRTP — resolved at compile time, zero overhead
template<typename Derived>
struct Strategy {
  void on_tick(Tick& t) {
    static_cast<Derived*>(this)->on_tick_impl(t);
  }
};
struct MyStrat : Strategy<MyStrat> {
  void on_tick_impl(Tick& t) { /* ... */ }
};

5. Lock-free & atomics3 techniques▼

SPSC ring buffer — the workhorse of HFT

You seestd::mutex-guarded queue between threads — lock contention shows in perf
template<typename T, size_t N>
class SPSCQueue {
  static_assert((N & (N-1)) == 0, "N must be power of 2");
  alignas(64) std::atomic<size_t> head_{0};
  alignas(64) std::atomic<size_t> tail_{0};
  T buf_[N];
public:
  bool push(const T& val) {
    size_t h = head_.load(std::memory_order_relaxed);
    size_t next = (h + 1) & (N - 1);
    if (next == tail_.load(std::memory_order_acquire)) return false;
    buf_[h] = val;
    head_.store(next, std::memory_order_release);
    return true;
  }
  bool pop(T& val) {
    size_t t = tail_.load(std::memory_order_relaxed);
    if (t == head_.load(std::memory_order_acquire)) return false;
    val = buf_[t];
    tail_.store((t + 1) & (N - 1), std::memory_order_release);
    return true;
  }
};

Acquire/release vs seq_cst — know the cost

You seeAll atomics using default memory_order_seq_cst — emits MFENCE on x86
// SLOW: seq_cst (default) — full memory fence, ~20ns
x.store(42); // implicitly seq_cst

// FAST: release on store, acquire on load — just compiler fence on x86
x.store(42, std::memory_order_release); // ~0ns on x86 (just MOV)
int v = x.load(std::memory_order_acquire); // ~0ns on x86 (just MOV)

Seqlock — readers never block, even during writes

You seeShared data read by many, written by one (e.g., top-of-book quote)
struct SeqLock {
  std::atomic<uint32_t> seq{0};
  TopOfBook data;
  void write(const TopOfBook& d) {
    seq.store(seq.load(std::memory_order_relaxed) + 1, std::memory_order_release);
    data = d;
    seq.store(seq.load(std::memory_order_relaxed) + 1, std::memory_order_release);
  }
  TopOfBook read() {
    TopOfBook copy;
    uint32_t s;
    do {
      s = seq.load(std::memory_order_acquire);
      if (s & 1) continue; // write in progress
      copy = data;
    } while (seq.load(std::memory_order_acquire) != s);
    return copy;
  }
};

7. Measurement & profiling2 techniques▼

rdtsc nano-timer — measure without syscalls

You seeNeed per-message latency, but clock_gettime adds 20-50ns
inline uint64_t rdtsc() {
  uint32_t lo, hi;
  asm volatile("rdtsc" : "=a"(lo), "=d"(hi));
  return (uint64_t)hi << 32 | lo;
}
// Calibrate: tsc_freq = measure cycles over known time
// Then: nanoseconds = (end - start) * 1e9 / tsc_freq

uint64_t t0 = rdtsc();
process_tick(tick);
uint64_t t1 = rdtsc();
latency_histogram.record(t1 - t0);

Latency histogram — p50/p99/p99.9 is what matters

You seeAverage looks fine but occasional spikes. Average hides jitter.
struct Histogram {
  std::array<uint64_t, 10000> buckets{}; // 1ns resolution
  void record(uint64_t ns) {
    if (ns < buckets.size()) ++buckets[ns];
  }
  uint64_t percentile(double p) {
    uint64_t total = 0;
    for (auto b : buckets) total += b;
    uint64_t target = total * p;
    uint64_t cum = 0;
    for (size_t i = 0; i < buckets.size(); ++i) {
      cum += buckets[i];
      if (cum >= target) return i;
    }
    return buckets.size();
  }
}; // report: p50, p99, p99.9, max
